# B端产品多方案构思器

用于在产品设计前期，快速输出多个差异明显的 B 端或 SaaS 产品方案思路，帮助产品经理在进入详细方案设计前完成方向比较与取舍。

## 适用场景

当用户出现以下诉求时，使用本 skill：

- 希望针对同一个问题快速得到多个不同的产品方案方向
- 还处在方案设计初期，需要先比较不同路线，再决定是否进入详细 PRD
- 希望做方案权衡、方向探索或替代方案对比
- 需要偏 B 端 / SaaS 的方案思路，而不是直接展开成完整需求文档

## 核心目标

围绕同一需求输出 `N` 个明显不同的产品方案思路，其中：

- 最少输出 2 个
- 最多输出 4 个
- 如果用户未指定 `N`，默认输出 3 个

这些方案必须在产品逻辑、交互方式、系统组织方式或管理策略上存在实质差异，不能只是同一思路的小改款。

## 提问规则

在生成方案前，只有当关键信息缺失时才向用户提问。

- 最多提问 3 轮
- 每轮最多 3 个问题
- 问题要短，尽量面向决策
- 如果可以合理假设，就不要追问

典型追问维度包括：

- 目标用户角色
- 业务场景或模块范围
- 目标结果或关键指标
- 是否存在成本、周期、组织偏好等约束

## 输出要求

最终结果必须使用 Markdown 分节形式逐个输出方案，不再使用表格。

推荐结构如下：

````markdown
## 方案1：XXX

- 设计理念：...
- 参考竞品 / 经典思路：...
- 原型示意图：[点击预览方案1原型](file:///absolute/path/to/原型-方案1-xxx.html) （请先使用工具将完整的 HTML 代码写入本地文件，然后在此处输出文件链接）
- 优势：...
- 劣势：...
- 预估工作量：...

## 方案2：XXX

- 设计理念：...
- 原型示意图：[点击预览方案2原型](file:///absolute/path/to/原型-方案2-xxx.html)
- 优势：...
- 劣势：...
- 预估工作量：...
````

补充要求：

- 仅输出 2 到 4 个方案
- 每个方案按固定字段逐个展开
- `设计理念`、`优势`、`劣势` 一般控制在 1 到 2 句话
- 每个方案都应增加 `预估工作量` 字段
- `预估工作量` 以人天表示，按 `1天 = 8小时` 计算
- 工作量可以使用范围表达，例如 `10-15人天`、`20-30人天`
- 优先对推荐方案给出更明确的预估，也可对全部方案分别估算
- 如果某个方案明显参考了具体竞品方案或行业经典设计思路，可增加 `参考竞品 / 经典思路` 字段
- `参考竞品 / 经典思路` 仅在有明确参考价值时输出，不要为了凑字段而强行标注
- 标注竞品时尽量说明参考的是哪类设计思路，不要只写产品名
- 表达要简洁，便于顺序阅读和横向对比
- 默认使用用户提问时的语言输出
- 除非用户明确要求，不要在方案前后补充额外说明

## 方案设计指引

每个方案都应体现 B 端 / SaaS 产品里常见、且可识别的不同设计模式。你可以参考典型企业产品的设计思路，例如 Salesforce、Workday、PeopleSoft、钉钉、飞书、北森、纷享销客、销售易、用友、金蝶、盖雅、肯耐珂萨、i人事、薪人薪事、2号人事部、有赞、微盟等产品。

注意：这些产品仅作为设计启发来源，不要把它们写成事实引用；如果不确定，不要声称某个产品一定采用了某种模式。

如果某个方案与具体竞品或行业经典设计思路高度相似，允许明确标识参考来源，例如：

- `参考竞品 / 经典思路：类似 Workday 的全球化主数据治理思路`
- `参考竞品 / 经典思路：类似 飞书 / 钉钉 的前台自助配置思路`
- `参考竞品 / 经典思路：行业常见的规则引擎 + 属地配置模式`

标识时遵循以下原则：

- 优先描述“参考了什么设计模式”，其次再写竞品名称
- 如果只是受到启发、但并不接近某个成熟方案，可以不标
- 如果已经存在行业经典方案或竞品成熟做法，优先将其作为推荐方向之一
- 只有在创新方案确实带来明显收益时，才把它放在经典方案之前

常见的差异化维度包括：

- 中台集中治理型 vs 前台自助配置型
- 规则驱动流程型 vs 向导引导型
- 表单配置型 vs 画布编排型
- 标准化优先型 vs 灵活性优先型
- 管理者视角型 vs 员工视角型 vs 管理员视角型
- 轻量快速上线型 vs 强治理企业级型
- 单模块闭环型 vs 跨模块协同型

## 差异性校验

输出前自行检查：

- 每个方案是否从不同角度解决同一个问题
- 优势和劣势是否避免机械重复
- 原型示意图是否体现方案特征，而不是泛泛描述
- 逐个输出的结构是否清晰统一，便于快速浏览和比较
- 工作量估算是否与方案复杂度相匹配，且推荐方案有明确人天判断

如果两个方案过于相似，应合并或替换其中一个。

## 工作量估算要求

`预估工作量` 用于帮助产品经理快速判断推荐方案的大致实施成本，默认按 `1天 = 8小时` 计算。

估算规则：

- 估算口径默认为产品设计与常规研发实现的粗粒度方案级预估，不要求拆到任务级
- 可以按单一总量表达，例如 `12人天`
- 也可以按范围表达，例如 `15-20人天`
- 若用户未指定角色口径，默认理解为单方案整体实现工作量的粗估
- 若信息不足，可在一句话内说明该估算基于常规中后台 Web 产品复杂度假设

估算时重点考虑：

- 是否新增复杂规则引擎、排班算法、流程配置等核心能力
- 是否涉及多角色、多页面、多状态流转
- 是否需要较多数据迁移、兼容或联动改造
- 是否依赖复杂报表、权限、审批、消息通知等配套能力

输出建议：

- 如果存在明显更推荐的方案，优先在该方案的 `预估工作量` 中给出更清晰的区间
- 如果方案差异较大，建议所有方案都给出各自的人天粗估，方便横向比较
- 避免给出过于精确的数字，例如 `17.5人天`

## 原型示意图要求

`原型示意图` 必须提供可以直接点击预览的 HTML 文件链接（本地地址），绝对不要在文本回答中直接输出长篇的前端代码块或字符画。

基本要求：

- 对于每个方案的原型，你需要主动使用文件写入工具（如 Write）将对应的原型代码保存为一个独立的本地 HTML 文件。
- 文件名建议为 `原型-方案X-XXX.html`。
- 生成文件后，在最终给用户的文本回答中，只需在 `原型示意图` 处提供该文件的 Markdown 链接（例如：`[预览方案1原型](file:///absolute/path/to/原型-方案1.html)`），以便用户直接在浏览器中打开预览。
- 每个 HTML 文件必须是包含 CDN 引用的完整文档（即从 `<!DOCTYPE html>` 到 `</html>`），推荐使用 **原生 HTML + Tailwind CSS**（以及按需引入 Mermaid.js）。
- 原型需要包含基础的布局、占位符、关键操作按钮和数据展示区域，能够直观体现出不同方案的交互结构差异。
- 绝对不要再使用 ASCII 简笔画（如 `+---+` 这种纯文本形式），也不要输出成堆的 `html` 或 `tsx` 代码块。

## 输出风格

- 优先采用务实、常见、可落地的企业产品设计模式
- 若行业已有经典设计思路或成熟竞品方案，优先推荐这些更稳妥的方向
- 创新方案可以输出，但应明确其额外收益与额外风险
- 平衡易用性、治理能力与实现成本
- 明确体现权衡关系
- 在推荐方案中体现大致工作量，帮助用户做投入产出判断
- 目标是快速比较，而不是展开成长篇方案

## 示例调用

用户示例：

`帮我给一个绩效管理场景输出3个完全不同的B端产品方案思路，方便我做前期比较。`

期望行为：

- 仅在场景过于模糊时追问
- 输出 3 个差异明显的方案
- 按方案 1、方案 2、方案 3 逐个输出
- 每个方案都包含可直接渲染或预览的原型代码（如 React/HTML/Mermaid）
- 如果方案明显借鉴经典做法，可标识参考竞品或经典思路
- 对推荐方案给出大致人天预估

## 调用示例模板

可直接复制以下模板给模型使用：

```markdown
请基于以下背景，输出 [N] 个完全不同的 B 端 / SaaS 产品方案思路，供前期方案比较使用。

背景信息：
- 业务场景：[例如：绩效管理 / 招聘协同 / 销售线索分配 / 门店会员运营]
- 目标用户：[例如：HRBP / 部门主管 / 销售经理 / 运营管理员]
- 核心目标：[例如：提升流程效率、加强过程管控、提高转化率、降低配置门槛]
- 已知约束：[例如：希望快速上线 / 研发资源有限 / 强调集团管控 / 需要兼顾灵活配置]

输出要求：
- 输出 [N] 个方案，其中最少 2 个，最多 4 个
- 各方案必须明显不同，不能只是同一思路的小改动
- 按 Markdown 分节逐个输出，不要用表格
- 每个方案只包含：方案名称、设计理念、原型示意图、优势、劣势、预估工作量
- 如果某个方案参考了具体竞品或行业经典设计思路，可以补充 `参考竞品 / 经典思路`
- 其中设计理念、优势、劣势每项尽量控制在 1 到 2 句话
- 原型示意图必须直接提供可点击预览的本地 HTML 文件链接（请先将生成的原型代码保存为独立的 `.html` 文件），绝对不要在回答中直接输出代码块或纯文本 ASCII 字符画
- 如果已有行业经典方案或成熟竞品方案，优先推荐这些更稳妥的方向
- `预估工作量` 按人天输出，按 1 天 = 8 小时计算
- 如果信息不足，可以先提问，但最多 3 轮、每轮最多 3 个问题
```

你也可以使用更简化的一句话版本：

```markdown
请围绕“[业务场景]”输出 3 个完全不同的 B 端产品方案思路，按 Markdown 分节逐个展示方案名称、设计理念、原型示意图、优势、劣势、预估工作量；如某个方案参考了具体竞品或行业经典思路，可标识出来；如已有成熟经典方案可优先推荐；对于原型示意图，请先使用工具将前端代码（如 HTML+Tailwind）写入独立的本地文件，然后直接在文本中提供该 HTML 文件的 Markdown 点击预览链接，切勿在对话中直接输出长篇代码块或纯文本字符画；`预估工作量` 按人天输出并按 1 天 = 8 小时计算，其他部分简洁表达；如有必要可先提问，但不要超过 3 轮、每轮不要超过 3 个问题。
```
