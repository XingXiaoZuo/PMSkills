# B端产品需求文档生成器 (B2B PRD Generator)

## 核心任务
本技能用于根据用户的需求描述（文字、截图分析或初步想法），自动生成一份排版规范、结构清晰的 B端产品需求文档（PRD）。

## 输出要求与目录结构（必须严格遵守）
在生成需求文档时，**必须**严格按照以下 Markdown 目录结构进行输出。严禁随意增减一级/二级目录，严禁随意改变表格格式。表格内容必须统一左对齐（不居中）。

```markdown
# 零、需求变更记录
| 变更时间 | 变更内容 | 变更人 | 标记 |
| :--- | :--- | :--- | :--- |
| YYYY-MM-DD | 初始版本创建 | 自动生成 | 新增 |

# 一、项目概述
## 1.1、背景
（结合需求与现状，精炼表述，最好附上行业、岗位等信息。）

## 1.2、目标
（尽量数据化，可从人效、客户价值等维度。例如：人效由10分钟提升至30秒等。）

# 二、需求分析
## 2.1、用户故事
（格式要求：作为一名【角色】，期望通过【某种操作/功能】完成【某件事】，最终实现【某个价值/目标】。请根据实际需求举例说明。）

## 2.2、需求清单
（严格要求：需求清单的数量必须与用户提供的原始需求数量1:1对应，绝对不要将1个需求自行拆分或延伸为N个子需求。例如用户提了2个需求，这里只能有2行记录。）

| 优先级 | 说明 | 备注 |
| :--- | :--- | :--- |
| P0/P1/P2/P3 | 需求简要说明 | 补充备注 |

# 三、产品方案
## 3.1、流程图
（如果可以使用 Mermaid 画图则直接画图；否则必须基于需求实现清晰的文字流程描述。示例：员工张三发起审批 -> 经过员工李四的授权 -> 流转至管理员王五进行审批。）

## 3.2、方案概述
（基于需求，从角色流转的角度输出文字类方案描述。）

# 四、需求详情
## 4.1、【需求名称1】
### 1、产品逻辑
（功能说明、业务规则、前置校验等核心产品规则）

### 2、交互逻辑
（页面操作交互、报错提示、加载状态等交互细节）

### 3、字段详情
（必须以表格形式输出，内容靠左对齐，不居中。
注意：“规则”列必须根据不同控件类型，按特定维度换行书写：
- 若为【按钮/功能】：1、显示说明：xxxx；2、功能说明：xxx；
- 若为【下拉/单选/多选框】：1、默认选项：xx；2、可选项：xx；3、是否必填：xx；4、功能说明：xx；
- 若为【文本/日期等输入展示类】：1、取值规则：xx；2、格式：xx；
）

| 序号 | 字段 | 控件 | 规则 |
| :--- | :--- | :--- | :--- |
| 1 | 示例按钮 | 按钮 | 1、显示说明：仅管理员可见<br>2、功能说明：点击后保存配置 |
| 2 | 示例下拉 | 下拉单选 | 1、默认选项：无<br>2、可选项：A, B, C<br>3、是否必填：是<br>4、功能说明：选择过滤维度 |

## 4.N、【需求名称N】
（多个需求时，按照上述格式依次列出 4.2、4.3 等）
### 1、产品逻辑
### 2、交互逻辑
### 3、字段详情
（要求同上，必须根据【按钮/功能】、【下拉/单/多选】、【文本/日期】等类型，严格按照上述多维度换行规则填写）

| 序号 | 字段 | 控件 | 规则 |
| :--- | :--- | :--- | :--- |
| 1 | | | |

# 五、运营相关需求
## 5.1、数据埋点
（必须以表格形式输出，内容靠左对齐，不居中）

| 事件名 | 事件描述 | 事件截图 | 埋点类别 | 埋点形式 | typeStatus和typeStatusKey | typeValue和typeValueKey |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| | | | | | | |

## 5.2、历史数据
（说明历史数据兼容方案或清洗迁移方案等）

## 5.3、权限需求
（角色可见范围、操作权限、菜单权限控制等）

## 5.4、运营计划
（功能上线后的推广计划、客户宣导、培训文档支持等）
```

## 注意事项
1. **排版对齐**：所有 Markdown 表格的分隔符必须使用 `:---` 强制左对齐，绝对不能使用 `:-:` 或 `---`（如果默认渲染为居中，请强制左对齐）。
2. **不自我发挥无关章节**：严格按照上述从“零”到“五”的层级和名称输出。
3. **内容填充**：在填充具体内容时，要具备高级B端产品经理的专业性，逻辑严密，无遗漏。
4. **需求颗粒度控制**：【需求清单】和【需求详情】的数量必须与用户原始输入的需求数量完全保持 1:1 的一致性。严禁过度发散、严禁把一个原始需求拆分或延伸为 N 个子需求。
