# B端需求工作量评估

遵循“需求是1，方案是0”的方法论，在不清楚需求的情况下，必须问清楚需求，需求清楚后，再确认解决方案，最后才是输出对应工作量。
核心目标：需求→场景→模块→功能→原子任务，拆解到「不可再拆分、可独立排期/估人天」的最小功能颗粒，直接支撑工作量评估、排期、开发认领。

## 一、核心拆解总原则（确保可估工作量）
1. 原子化原则：一个功能点只做一件事，不混合多个逻辑；
2. 闭环原则：包含「新增/查询/编辑/删除/校验/联动/异常/日志」完整链路；
3. 边界清晰：区分前端、后端、数据、配置、对接、产品、测试；
4. 可量化：每个拆解项能直接估：人天、难度、依赖、优先级。

## 二、标准拆解五步方法论（直接套用）

### 第一步：需求定性&范围锁定（防范围蔓延）
1. 明确核心目标：这个需求解决什么问题？
2. 划定使用角色：管理员/员工/HR/第三方用户
3. 限定业务边界：哪些做、哪些不做、兼容旧逻辑吗？
作用：避免拆解中无限加功能，减少无效工作量评估。

### 第二步：按「用户场景」拆解（业务维度拆分）
把大需求拆成独立使用场景，场景是拆解第一层颗粒：
- 例：【员工档案优化】
  - 场景1：员工手动新增档案
  - 场景2：档案信息编辑修改
  - 场景3：档案批量导入/导出
  - 场景4：档案字段权限控制
  - 场景5：档案删除&回收站

### 第三步：场景拆「业务模块」（系统维度拆分）
每个场景下，拆分系统固定模块，参考SaaS通用模块：
- 基础操作：列表、详情、新增、编辑、删除、筛选、搜索、分页
- 规则逻辑：数据校验、必填、唯一值、联动、计算公式、状态流转
- 权限体系：字段权限、按钮权限、数据权限、角色控制
- 数据能力：导入、导出、打印、脱敏、归档、日志
- 异常兜底：报错提示、空页面、数据兜底、重试机制
- 对接联动：同步其他模块、接口对接、消息通知

### 第四步：模块拆「最小功能点」（核心落地层，用于估工作量）
关键：拆到开发最小交付单元，颗粒度标准：
一个功能点 = 单一操作+单一逻辑，1个后端接口/1块前端页面组件可完成
拆解公式：页面/位置 + 操作行为 + 业务规则/特殊逻辑

**举例示范（HR SaaS真实案例）**
原始需求：实现「部门维度考勤汇总报表」
1. 前端功能点
   - 考勤报表新增【部门汇总】标签页
   - 部门报表：按组织架构树形筛选部门
   - 报表支持时间维度筛选（日/周/月）
   - 展示字段：应出勤、实际出勤、迟到、早退、旷工、请假合计
   - 报表列表支持单列排序
   - 部门汇总报表数据导出Excel
   - 空数据、无权限兜底页面展示
2. 后端功能点
   - 开发部门考勤汇总查询接口
   - 多维度时间条件过滤逻辑封装
   - 迟到/早退/旷工数据聚合计算逻辑
   - 部门组织数据权限过滤
   - 报表导出文件生成&下载接口
   - 数据查询性能优化、分页逻辑
3. 附属配套功能点
   - 新增操作日志：报表查询、导出记录
   - 角色权限配置：部门报表查看按钮权限
   - 产品：新增报表页面逻辑设计、需求宣讲、测试验收

### 第五步：功能点拆「任务项」（最终用于工作量评估）
把单个功能点，再按岗位职责拆分，直接用来估人天：
- 产品：需求梳理、原型设计、逻辑定义、验收把关
- 前端：页面开发、组件封装、交互逻辑、表单校验
- 后端：表结构调整、接口开发、逻辑计算、事务处理
- 数据：数据迁移、历史数据兼容、定时任务
- 测试：用例编写、功能测试、边界测试、回归测试
- 配置/运维：环境配置、参数配置、菜单配置

## 三、输出规范与模板（严格遵守）

**最终输出必须是以下Markdown格式的表格：**

| 场景 | 功能点详情 | 后端工作量 | 前端工作量 |
| ---- | ---------- | ---------- | ---------- |
| 示例场景：部门报表管理 | 1. [前端] 部门树形筛选交互及报表页面展示<br>2. [后端] 报表数据多维度查询及分页接口<br>3. [后端] 报表导出文件生成及下载 | 2.5天 | 1.5天 |
| 示例场景：附属配套功能 | 1. [后端] 操作日志记录<br>2. [后端] 权限配置 | 1.0天 | 0天 |
| **测试工作量** | - | **1.0天** | - |
| **产品工作量** | - | **1.0天** | - |
| **汇总** | - | **x.x天** | **x.x天** |

**评估规则说明：**
1. **列维度**：列只包含“场景”、“功能点详情”、“后端工作量”、“前端工作量”。
2. **行维度**：前半部分每一行对应第二步拆解的一个“用户场景”。
3. **测试与产品行**：不再按场景拆分，测试和产品作为独立的两行放在场景行之后。
   - **测试工作量**：一般按“后端总工作量的1/3”进行评估计算（取0.5的整数倍）。
   - **产品工作量**：按需求整体复杂度分为五级（简单0.5-1天、中等1-2天、复杂2-4天、非常复杂4-6天），且**一般不能超过后端工作量的1/4**。
   - **注意**：最终输出的表格里，“测试工作量”、“产品工作量”的“功能点详情”列直接填“-”，**绝对不要**把评估标准输出在表格中。
4. **插件开发模式（特殊规则）**：当用户明确说明需求采用“插件”或“插件方式”实现时：
   - 前端工作量默认评估为 **0天**（无UI交互）。
   - 后端工作量应比正常系统功能开发至少**减少 1/2**（因复用了插件底层框架与标准出入参）。
   - 测试和产品工作量依然按后端工作量折算。
   - 若未明确提及“插件”，则按正常功能评估。
5. **详情维度**：功能点详情采用序号说明，将同一个场景下的N个前后端功能点写在一起。
6. **最小颗粒度**：所有工作量评估的最小颗粒度为 **0.5天**。
7. **极简拆解公式**（快速分析阶段用）：需求 = N个使用场景 + 每个场景=页面操作+数据逻辑+权限控制+导出/通知+异常处理

## 四、避坑：拆解过粗/过细的问题
1. 拆太粗：一个功能点包含5个逻辑 → 评估工时不准、开发扯皮、排期混乱
2. 拆太细：按钮点击、文字修改都算功能 → 管理成本极高，没必要
3. 最优颗粒度：
   - 业务功能：按「一个独立操作+一套独立逻辑」拆分
   - 最小颗粒度：所有评估最小必须为 **0.5人天/个**
   - 复杂逻辑功能：1~3人天/个

## 五、配套：工作量快速评估方法（和拆解联动）
1. 按拆解后的最小功能点逐个估时
2. 区分：新增开发、改造旧逻辑、对接第三方、数据迁移
3. 叠加系数：
   - 复杂业务规则 ×1.5
   - 跨模块联动 ×1.3
   - 历史数据兼容 ×1.2
4. 汇总：前端工时+后端工时+测试+产品+配置 = 整体需求工作量

## 六、补充：高阶拆解方法论（复杂业务用）
1. DDD领域驱动拆解：按业务域→子域→业务实体→行为拆解，适合大型SaaS
2. 用户旅程拆解：按用户完整操作链路：进入→操作→提交→结果→后续通知
3. 四象限拆解：正向流程、逆向流程、异常流程、兼容流程全覆盖
