# 高保真原型还原器 (Exact Match Prototype Generator)

用于根据用户提供的“需求文档”或“参考截图 + 需求说明”，直接输出与参考图在视觉风格、主题颜色、交互形式上 **100% 保持一致** 的前端原型代码。

## 适用场景

当用户出现以下诉求时，使用本 skill：
- 提供了一张竞品截图或设计草图，要求“按这个风格画一个原型”
- 提供了一份 PRD 需求文档，并附带了视觉参考规范，要求生成高保真界面代码
- 强调需要“100%还原”、“颜色/排版一致”的 UI 原型开发

## 核心还原规则：100% 视觉与交互一致

当用户提供了参考截图或明确了设计规范时，必须遵循以下严格的还原原则：

1. **色彩提取**：精确提取截图中的主色调（Primary Color）、辅助色（Secondary Color）、背景色（Background）、文本色（Text）和边框色（Border）。在代码中使用最接近的 Tailwind 颜色类，或直接使用自定义 HEX/RGB 值（例如 `bg-[#1E88E5]`）。
2. **排版与布局**：严格遵守截图中的布局模式（如侧边栏宽度、顶部导航高度、卡片间距、栅格比例）。对齐方式（Flex/Grid）、内边距（Padding）和外边距（Margin）必须与原图视觉比例完全一致。
3. **字体与排版**：还原标题、正文、辅助说明文本的字号大小（Text Size）、字重（Font Weight）以及文字颜色层级。
4. **交互形式**：还原截图中的交互元素形式（例如：是弹窗还是抽屉？是下拉菜单还是平铺选项？是开关还是复选框？）。如果截图中有悬浮态（Hover）或选中态（Active/Focus）的暗示，需在代码中实现对应的交互样式（例如 `hover:bg-gray-100`）。
5. **图标与占位符**：合理使用通用的图标库（如内联 SVG、Heroicons 等）来还原截图中的图标含义。如果有图片占位需求，使用背景色块或占位图服务。

## 默认视觉规范（XRXS / 薪人薪事风格基准）

如果用户要求“参考我提供的截图风格”且处于类似上述 HR SaaS 场景，必须默认应用以下从截图中提取的视觉特征：

- **主色调 (Primary Color)**: 经典薪人薪事绿（使用自定义颜色 `[#00b38a]`）。主要操作按钮（如“添加”、“确定”）、选中态的文字、单选框/复选框激活态、关键图标等均使用此颜色（如 `bg-[#00b38a]`、`text-[#00b38a]`）。
- **页面框架 (Layout)**:
  - **顶部导航栏 (Top Nav)**: 白色背景，高度约 `h-14`，底部带极浅灰边框。左侧包含企业下拉与 Logo，中间包含搜索框（圆角、灰色图标），右侧为应用中心、通知、反馈、设置及用户头像。
  - **左侧菜单 (Sidebar)**: 极简白色背景。一级菜单带有左侧细线标识和图标；选中状态的菜单项文字变为绿色 `text-[#00b38a]`。
  - **主内容区 (Main Content)**: 整体背景常为浅灰色（如 `bg-[#f5f7fa]`），内部功能区以白色卡片（`bg-white`）承载，卡片带微弱阴影或浅灰边框。
- **组件细节 (Components)**:
  - **表格 (Table)**: 扁平化风格。表头背景为极浅灰或纯白，文字偏灰（如 `text-gray-500`）；行与行之间仅有浅灰色水平分割线（`border-b border-gray-100`），无垂直边框。
  - **表单与输入框 (Form)**: 输入框默认带浅灰色边框，聚焦时边框和发光环变为主色绿（`focus:border-[#00b38a] focus:ring-[#00b38a]`）。
  - **标签与状态 (Tabs & Tags)**: 页面内导航常采用顶部 Tab 切换（如下划线选中态，底边框为绿色 `border-[#00b38a]`）；状态标识（如“固定班次”）使用浅绿色背景加深绿色文字组合。
  - **卡片式表单布局**: 在如“添加班次”这种页面中，表单按“基础信息”、“考勤时间”分块，每个块左上角有绿色的短竖线装饰符作为标题引导。

## 输出要求

- 默认生成 **原生 HTML + Tailwind CSS** 结构的高保真代码。
- **核心动作要求：必须直接创建文件！** 在生成代码后，必须主动使用文件写入工具（如 `Write`），将完整代码直接写入到当前工作目录下的一个本地 `.html` 文件中（例如命名为 `原型-页面名称.html`）。
- **禁止纯文本输出**：绝对不要仅仅在对话框中输出一段巨大的 Markdown 代码块让用户自己去复制，必须帮用户把物理文件建好。
- **提供预览指引**：写入文件成功后，在回复中给出该文件的本地绝对路径，并提示用户可以双击或在浏览器中直接打开预览。
- **代码完整性**：写入的 HTML 文件必须包含从 `<!DOCTYPE html>` 到 `</html>` 的完整结构，并引入 Tailwind CSS 的 CDN。绝对不要省略 CSS 类名或关键布局代码，不要使用 `<!-- 此处省略 -->` 或 `...` 占位符。
- 输出前仔细对比需求说明和提取的截图特征，确保业务逻辑没有遗漏，视觉规范没有走样。

## 提问规则

在生成方案前，如果关键信息缺失，可以向用户提问：

- 如果用户仅提供了截图，但没有提供具体要实现的功能需求说明，需提问确认：“请问您需要基于这个风格实现什么具体的业务功能？还是仅仅一比一还原这张图？”
- 如果用户仅提供了需求文档，但没有提供截图，需提问确认：“请问您是否有期望参考的视觉风格截图？如果没有，我将使用标准的现代 B 端/SaaS 风格为您绘制。”
- 提问最多不超过 2 轮，每次问题要简短直接。

## 调用示例模板

用户可以直接复制以下模板使用：

```markdown
请基于我提供的 [需求说明] 以及附带的 [参考截图]，为我绘制完整的产品原型代码。

要求：
1. 整体原型风格、交互形式、主题颜色必须 100% 与我提供的截图保持一致。
2. 请使用 原生 HTML + Tailwind CSS 编写代码，并直接为我生成一个 .html 文件供我预览，禁止只在对话框中发代码块。
3. 确保需求说明中的所有字段和操作入口均在界面中体现。
4. 不得省略任何布局和样式代码。
```
